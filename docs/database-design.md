# 二手交易平台数据库设计说明

## 1. 文档说明

本文档基于当前项目后端代码整理，介绍数据库中各张表的作用、主要字段的含义，以及表之间的关系。

本文档的信息来源主要有三类：

1. MyBatis Mapper SQL
2. `DatabaseMigrationService` 中的自动建表与补字段逻辑
3. `PO / VO / Repository` 中对字段的读写方式

其中：

- `user_address`、`favorite_goods`、`chat_message`、`goods_category`、`wanted_post` 的结构可以直接从迁移脚本中确认。
- `salegoods`、`shoppingcart` 等早期表没有完整建表 SQL 保存在当前仓库中，但可以从 Mapper 的增删改查语句中明确推断结构。

---

## 2. 数据库整体说明

当前数据库围绕“校园二手交易”业务展开，主要包含以下几类数据：

1. 用户数据
2. 商品数据
3. 交易与订单数据
4. 购物车与收藏数据
5. 聊天与求购数据
6. 地址与分类数据

数据库中的关系大体可以理解为：

- 一个用户可以发布多个商品
- 一个用户可以收藏多个商品
- 一个用户可以把多个商品加入购物车
- 一个订单连接一个买家、一个卖家和一个商品
- 一个用户可以维护多个收货地址
- 一个用户可以发布多个求购信息

当前项目主要依赖应用层维护这些关系，代码中没有看到显式的数据库外键约束定义。

## 4. 各数据表说明

## 4.1 `user`

### 表作用

用于保存平台用户账号、联系方式、身份状态和个人资料。

该表同时保存：

- 普通用户
- 管理员用户

系统通过 `UID` 前缀区分角色：

- `NORM` 开头：普通用户
- `ADMI` 开头：管理员

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `UID` | 用户唯一编号 | 主业务标识；普通用户通常为 `NORM + 学号` |
| `Uname` | 用户名 | 登录后页面显示的昵称或姓名 |
| `Email` | 邮箱 | 注册和个人资料中使用 |
| `PhoneNum` | 手机号 | 用户联系方式 |
| `Password` | 密码 | 保存 MD5 后的密码值 |
| `Status` | 用户状态 | `0=正常`，`1=需修改密码`，`2=已停用` |
| `StudentNo` | 学号 | 普通用户注册时的重要身份字段 |
| `RealName` | 真实姓名 | 个人资料字段 |
| `CampusVerified` | 校园认证标记 | `1` 表示已认证，`0` 表示未认证 |
| `Avatar` | 头像地址 | 可选字段 |
| `Bio` | 个人简介 | 可选字段 |

### 与业务的关系

- 登录、注册、资料修改、管理员用户管理都依赖这张表。
- 商品发布前会检查 `CampusVerified`。
- 管理员停用用户时会修改 `Status`。

---

## 4.2 `marketgoods`

### 表作用

用于保存平台商品主体信息，是商品浏览、商品详情、发布商品、审核商品的核心数据表。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `GID` | 商品编号 | 商品主业务 ID |
| `Name` | 商品名称 | 例如教材、耳机、台灯 |
| `Kind` | 商品分类 | 对应分类名称 |
| `Price` | 商品价格 | 单价 |
| `Number` | 商品库存 | 当前剩余数量 |
| `Image` | 商品图片路径 | 可以是相对路径或完整地址 |
| `Comment` | 商品描述 | 卖家填写的商品说明 |
| `Status` | 商品状态 | `PENDING_REVIEW`、`ACTIVE`、`REJECTED`、`OFF_SHELF`、`BANNED` |
| `DeliveryMode` | 交付方式 | `SELF_PICKUP` 或 `CAMPUS_DELIVERY` |
| `PickupLocation` | 自提点/送货说明 | 交付地点补充说明 |
| `CampusOnly` | 是否仅校内交易 | 当前代码中通常默认 `1` |
| `ReviewNote` | 审核备注 | 管理员审核商品时填写 |
| `PublishedAt` | 发布时间 | 字符串形式的发布时间 |

### 状态字段说明

| 状态值 | 含义 |
| --- | --- |
| `PENDING_REVIEW` | 待管理员审核 |
| `ACTIVE` | 已上架，可被浏览和购买 |
| `REJECTED` | 审核被驳回 |
| `OFF_SHELF` | 已下架或库存为 0 |
| `BANNED` | 违规下架 |

### 与业务的关系

- 商品广场读取这张表中的可上架商品。
- 卖家发布和编辑商品时写入这张表。
- 下单时会扣减 `Number`。
- 管理员审核时会修改 `Status` 和 `ReviewNote`。

---

## 4.3 `salegoods`

### 表作用

用于维护“哪个用户发布了哪个商品”的关系，是卖家与商品之间的关联表。

这张表的完整建表 SQL 当前仓库中未保留，但从现有 Mapper 可以明确看出它至少包含以下字段。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `UID` | 卖家用户 ID | 对应 `user.UID` |
| `GID` | 商品 ID | 对应 `marketgoods.GID` |

### 与业务的关系

- 查询某件商品属于哪个卖家：`WhoseGoods`
- 查询某个卖家发布的全部商品：`ShowGoods`
- 发布商品时插入一条卖家-商品关联
- 删除商品时删除关联

### 表特点

- 这是一个典型的关系映射表。
- 逻辑上可以理解为“一个用户对多个商品的一对多关系”。

---

## 4.4 `shoppingcart`

### 表作用

用于保存用户加入购物车但尚未下单的商品记录。

该表完整建表 SQL 当前仓库中未保留，但从 `CarDao.xml` 可以明确推断结构。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `UID` | 用户 ID | 购物车所属用户 |
| `GID` | 商品 ID | 加入购物车的商品 |
| `Number` | 购物车数量 | 用户想购买的数量，不是商品总库存 |

### 与业务的关系

- 购物车列表读取该表。
- 加入购物车时插入或更新该表。
- 从购物车下单后，会删除对应购物车记录。
- 商品被永久删除时，也会清理该表中的关联记录。

### 特别说明

- `shoppingcart.Number` 表示购物车中的购买数量。
- `marketgoods.Number` 表示商品库存。
- 这两个字段语义不同，前端购物车页会同时读取两者做比较。

---

## 4.5 `traderecord`

### 表作用

用于保存订单和交易记录，是平台买卖流程的核心表。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `PID` | 订单编号 | 订单主业务 ID |
| `BuyerID` | 买家用户 ID | 对应 `user.UID` |
| `Date` | 下单日期 | 订单日期部分 |
| `Time` | 下单时间 | 订单时间部分 |
| `GID` | 商品 ID | 对应商品编号 |
| `SellerID` | 卖家用户 ID | 对应卖家 |
| `Gname` | 商品名称快照 | 下单时商品名称 |
| `Gkind` | 商品分类快照 | 下单时商品分类 |
| `Gprice` | 商品价格快照 | 下单时商品单价 |
| `Gnumber` | 购买数量 | 本次购买数量 |
| `IsSent` | 是否已发货 | 布尔/标记字段 |
| `IsGot` | 是否已收货 | 布尔/标记字段 |
| `Status` | 订单状态 | 当前订单流程状态 |
| `DeliveryMode` | 交付方式 | 自提或送货 |
| `PickupLocation` | 自提/送货说明 | 交易地点说明 |
| `AppointmentTime` | 预约时间 | 当前实现中通常记录生成订单时的时间 |
| `Remark` | 订单备注 | 如“来自购物车下单”、“立即下单” |
| `AddressSnapshot` | 地址快照 | 下单时保存的地址文本快照 |

### 状态字段说明

| 状态值 | 含义 |
| --- | --- |
| `PENDING_PICKUP` | 待自提 |
| `PENDING_SHIPMENT` | 待发货 |
| `PENDING_RECEIPT` | 待收货 |
| `COMPLETED` | 已完成 |

### 与业务的关系

- 用户下单时向该表插入订单记录。
- 买家查看购买记录、卖家查看销售记录都依赖该表。
- 卖家发货、买家确认收货会更新该表的 `Status`、`IsSent`、`IsGot`。
- 删除订单时会删除该表记录。

### 特别说明

- 商品相关字段使用了“快照”思路，例如 `Gname`、`Gprice`、`AddressSnapshot`。
- 这样即使后续商品信息或地址信息发生变化，历史订单仍能保持原样。

---

## 4.6 `user_address`

### 表作用

用于保存用户的收货地址。

这张表由 `DatabaseMigrationService` 自动创建。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `Id` | 地址主键 | 自增主键 |
| `UID` | 所属用户 ID | 对应 `user.UID` |
| `ReceiverName` | 收货人姓名 | 地址联系人 |
| `PhoneNumber` | 联系电话 | 收货联系电话 |
| `CampusArea` | 校区/宿舍区 | 校园内区域 |
| `DetailAddress` | 详细地址 | 楼栋、寝室等详细信息 |
| `IsDefault` | 是否默认地址 | `1` 默认，`0` 非默认 |
| `CreatedAt` | 创建时间 | 地址创建时间 |

### 与业务的关系

- 个人中心可以新增、删除、修改地址。
- 下单时系统会从该表读取默认地址或指定地址，并生成订单地址快照。

---

## 4.7 `favorite_goods`

### 表作用

用于保存用户收藏的商品。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `Id` | 收藏记录主键 | 自增主键 |
| `UID` | 收藏用户 ID | 谁收藏了商品 |
| `GID` | 商品 ID | 收藏的是哪件商品 |
| `CreatedAt` | 收藏时间 | 创建时间 |

### 约束说明

- 建有唯一约束：`uk_favorite_uid_gid (UID, GID)`

这表示同一个用户不能重复收藏同一件商品。

### 与业务的关系

- 收藏列表依赖这张表。
- 商品永久删除时会清理对应收藏记录。

---

## 4.8 `chat_message`

### 表作用

用于保存用户之间的私信消息。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `Id` | 消息主键 | 自增主键 |
| `SenderUID` | 发送者用户 ID | 发消息的人 |
| `ReceiverUID` | 接收者用户 ID | 收消息的人 |
| `GoodsID` | 关联商品 ID | 可为空，表示消息与某商品相关 |
| `OrderPID` | 关联订单 ID | 可为空，表示消息与某订单相关 |
| `Content` | 消息内容 | 文本消息 |
| `CreatedAt` | 发送时间 | 消息创建时间 |
| `IsRead` | 是否已读 | `1` 已读，`0` 未读 |

### 与业务的关系

- 消息页的会话列表和聊天记录都来自这张表。
- 售前咨询通常通过 `GoodsID` 聚合。
- 售后沟通通常通过 `OrderPID` 聚合。
- 删除商品时，会把相关消息中的 `GoodsID` 置空，而不是直接删消息。

---

## 4.9 `goods_category`

### 表作用

用于保存商品分类配置，支持前台筛选和后台分类管理。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `Id` | 分类主键 | 自增主键 |
| `Code` | 分类编码 | 稳定标识，如 `TEXTBOOK` |
| `Label` | 分类名称 | 页面展示用，如“教材” |
| `SortOrder` | 排序值 | 用于控制分类显示顺序 |
| `Enabled` | 是否启用 | `1` 启用，`0` 禁用 |

### 约束说明

- 建有唯一约束：`uk_goods_category_code (Code)`

### 与业务的关系

- 前台商品分类筛选读取启用状态的分类。
- 发布商品时分类下拉框也来自该表。
- 管理员可以在后台新增或修改分类。

### 默认数据

系统启动时会自动初始化这些分类：

- `TEXTBOOK` 教材
- `DIGITAL` 数码
- `DORMITORY` 宿舍用品
- `SPORTS` 运动器材
- `DAILY` 日用品
- `OTHER` 其他

## 4.10 `wanted_post`

### 表作用

用于保存用户发布的求购信息。

### 主要字段

| 字段名 | 含义 | 说明 |
| --- | --- | --- |
| `Id` | 求购主键 | 自增主键 |
| `UID` | 发布用户 ID | 求购发布者 |
| `Title` | 求购标题 | 想买什么 |
| `Category` | 求购分类 | 所属商品分类 |
| `Budget` | 预算金额 | 可为空时默认 0 |
| `Keyword` | 关键词 | 用于匹配相关商品 |
| `Description` | 详细说明 | 求购补充信息 |
| `Status` | 求购状态 | 当前实现默认 `OPEN` |
| `CreatedAt` | 发布时间 | 创建时间 |

### 状态说明

当前代码中主要使用：

| 状态值 | 含义 |
| --- | --- |
| `OPEN` | 进行中的求购 |

前端枚举中还预留了 `CLOSED`，说明后续可扩展为关闭求购。

### 与业务的关系

- 求购广场展示所有开放中的求购信息。
- 用户可以查看自己发布的求购并删除。
- 查询求购时，系统会用 `Category` 和 `Keyword` 去匹配平台中的在售商品。

## 5. 表之间的关系说明

## 5.1 用户与商品

- `user.UID` <-> `salegoods.UID`
- `salegoods.GID` <-> `marketgoods.GID`

含义：

- 一个用户可以发布多个商品。
- `salegoods` 负责把“卖家”和“商品”关联起来。

## 5.2 用户与购物车

- `shoppingcart.UID` <-> `user.UID`
- `shoppingcart.GID` <-> `marketgoods.GID`

含义：

- 一个用户的购物车中可以有多个商品。

## 5.3 用户与收藏

- `favorite_goods.UID` <-> `user.UID`
- `favorite_goods.GID` <-> `marketgoods.GID`

含义：

- 一个用户可以收藏多个商品。

## 5.4 用户与订单

- `traderecord.BuyerID` <-> `user.UID`
- `traderecord.SellerID` <-> `user.UID`
- `traderecord.GID` <-> `marketgoods.GID`

含义：

- 一条订单记录连接一个买家、一个卖家和一个商品。

## 5.5 用户与地址

- `user_address.UID` <-> `user.UID`

含义：

- 一个用户可以维护多个地址。

## 5.6 用户与消息

- `chat_message.SenderUID` <-> `user.UID`
- `chat_message.ReceiverUID` <-> `user.UID`
- `chat_message.GoodsID` <-> `marketgoods.GID`
- `chat_message.OrderPID` <-> `traderecord.PID`

含义：

- 用户之间可以围绕商品或订单进行聊天。

## 5.7 用户与求购

- `wanted_post.UID` <-> `user.UID`

含义：

- 一个用户可以发布多个求购信息。

## 7. 数据库设计特点总结

从当前项目代码来看，数据库设计有以下几个特点：

1. 以 `user`、`marketgoods`、`traderecord` 为核心主表。
2. 用 `salegoods`、`shoppingcart`、`favorite_goods` 这类关系表扩展交易行为。
3. 订单表中保存了商品快照和地址快照，保证历史记录稳定。
4. 地址、聊天、求购、分类等功能采用新增表扩展，结构清晰。
5. 当前很多表关系由应用层维护，适合课程项目快速迭代，但数据库层面的约束还可以继续加强。

---

## 8. 结论

当前二手交易平台数据库已经覆盖了一个完整校园二手交易系统所需的核心数据：

- 用户
- 商品
- 卖家关系
- 购物车
- 订单
- 地址
- 收藏
- 聊天
- 分类
- 求购

这些表共同支撑了平台的主要业务闭环：  
用户注册登录 -> 浏览商品 -> 发布商品 -> 加购/收藏 -> 下单交易 -> 聊天沟通 -> 地址管理 -> 管理员审核与分类维护。

如果后续继续扩展系统，比较自然的方向包括：

- 增加数据库外键与索引说明
- 增加评价表
- 增加退款/取消订单表或更完整的订单状态流转
- 增加商品浏览记录、消息会话表等更细的数据模型
